# Core Concepts

## Status Model

```
open тЖТ in_progress тЖТ closed
  тЖУ         тЖУ          тЖУ
blocked   cancelled   (reopen тЖТ open)
  тЖУ
deferred
  тЖУ
  (auto-unblock тЖТ open when all blockers terminal)
```

### Statuses

| Status | Icon | Description |
|--------|------|-------------|
| `open` | `[ ]` | Ready to be worked on |
| `in_progress` | `[>]` | Actively being worked on (claimed) |
| `blocked` | `[!]` | Waiting on a dependency |
| `deferred` | `[~]` | Postponed for later |
| `closed` | `[x]` | Completed |
| `cancelled` | `[-]` | Abandoned |

### Terminal States

`closed` and `cancelled` are terminal. Use `IsTerminal()` to check тАФ never compare directly with `== StatusClosed`.

### Auto-Behaviors

- **Auto-unblock:** When a blocker reaches a terminal state, all dependents with all blockers terminal transition from `blocked` тЖТ `open`. Runs within the same transaction as the close/cancel.
- **Auto-progress parent:** When a child is claimed (тЖТ `in_progress`), the parent automatically moves from `open` тЖТ `in_progress`.
- **Close blocking:** A parent cannot be closed if any child is non-terminal. Close or cancel all children first.
- **Cascade cancel:** Cancelling a parent cascades to all non-terminal children.

## Task Hierarchy

### Ladder IDs

IDs encode the hierarchy:

```
st-a3f8              тЖР root task (epic)
тФЬтФАтФА st-a3f8.1        тЖР 1st child
тФВ   тФЬтФАтФА st-a3f8.1.1  тЖР grandchild
тФВ   тФФтФАтФА st-a3f8.1.2
тФФтФАтФА st-a3f8.2        тЖР 2nd child
```

Child IDs are generated by counting existing children: `parentID + "." + (count + 1)`.

### Hierarchy Behaviors

| Action on Parent | Effect on Children |
|-----------------|-------------------|
| Close | Blocked if any child non-terminal |
| Cancel | Cascades to all non-terminal children |
| Delete | Recursively deletes all descendants (depth-first) |
| Child claims | Parent auto-progresses to in_progress |

## Dependency DAG

### Dependency Types

| Type | Purpose | Affects Ready/Blocked? |
|------|---------|----------------------|
| `blocks` | Task A blocks Task B (B depends on A) | Yes |
| `relates_to` | Informational link | No |
| `duplicates` | B is a duplicate of A | No |

Only `blocks` type participates in auto-unblock and Ready/Blocked queries.

### Cycle Detection

Before adding any dependency, BFS is run from the target back to the source. If the source is reachable, the dependency would create a cycle and is rejected.

Full-graph cycle detection (DFS with in-stack tracking) is available via `DetectCycles()` for audit purposes.

### Ready and Blocked

- **Ready:** Open tasks with no unresolved `blocks` dependencies. All blockers must be terminal (closed or cancelled).
- **Blocked:** Tasks with status `blocked`.

## Claim System

### Atomic Claim

`Claim()` atomically sets assignee and status to `in_progress`:

```sql
UPDATE tasks SET assignee = ?, status = 'in_progress'
WHERE id = ? AND (assignee IS NULL OR assignee = '' OR assignee = ?)
```

If another agent has already claimed the task, `ErrAlreadyClaimed` is returned. The original claim is not overwritten.

### Re-Claiming

The same agent can re-claim its own task (the `OR assignee = ?` clause allows this).

### Release

`ReleaseClaim()` sets assignee to empty and status back to `open`.

### Lease Expiry

`ExpireLeases()` auto-releases claims where `updated_at` is older than the configured lease timeout (default: 30 minutes). This handles crashed agents that never release their claims.

## Event System

### Three Layers

1. **Events table** (always) тАФ append-only audit log in SQLite. Every mutation recorded.
2. **SDK callbacks** (opt-in) тАФ `store.On(EventType, func(Event))`. Fires synchronously after the DB write.
3. **Shell hooks** (opt-in) тАФ commands in `slate.yaml`. Fires as async goroutines.

### Event Types

| Type | When |
|------|------|
| `created` | Task created |
| `updated` | Field changed (title, description, priority, etc.) |
| `status_changed` | Status transition |
| `commented` | Comment added |
| `assigned` | Assignee changed |
| `closed` | Task closed |
| `dependency_added` | Dependency created |
| `dependency_removed` | Dependency removed |
| `deleted` | Task deleted (emitted to listeners only, not stored) |

### Hook Configuration

```yaml
hooks:
  on_status_change:
    - command: "notify.sh {id} {new}"
      filter:
        new_status: "closed"
```

Template variables: `{id}`, `{old}`, `{new}`, `{actor}`, `{field}`

Filter keys: `new_status`, `old_status`, `assignee`

## Custom Attributes

### Two-Layer System

**Layer 1 тАФ Definitions (schema registry):**
- Must define a key before setting values
- Types: `string`, `boolean`, `object` (JSON)
- Idempotent: defining an existing key is a no-op

**Layer 2 тАФ Values (per-task data):**
- FK constraint: key must exist in definitions
- Type validation on write:
  - `boolean`: must be `"true"` or `"false"`
  - `object`: must be valid JSON
- Upsert semantics (INSERT OR CONFLICT)

### Attribute Methods

```go
store.DefineAttr(ctx, "env", slate.AttrString, "Environment")
store.SetAttr(ctx, taskID, "env", "production")
attr, _ := store.GetAttr(ctx, taskID, "env")
fmt.Println(attr.StringValue()) // "production"
```

## Metrics

### Cycle Time

Duration from task creation to close: `closed_at - created_at`.

### Throughput

Count of tasks closed in a time range, derived from events table.

### Next

`Next()` suggests the highest-impact ready task by counting transitive dependents (BFS). The task that unblocks the most downstream work is recommended first.

## Notion Sync

### Bidirectional Sync Model

Slate syncs with Notion bidirectionally: local changes push to Notion, Notion changes pull to Slate. Sync is **polling-based** (on-demand via `slate notion sync`).

### Field-Level Merge

Changes to different fields don't conflict. A PM can change priority in Notion while an agent updates status in Slate тАФ both changes apply cleanly.

### Configurable Mapping

Real Notion databases have different schemas. Slate uses configurable mappings (`notion.yaml`) for:
- **Property names**: which Notion column maps to which Slate field
- **Status values**: many-to-one mapping (e.g., "In Review" + "On QA" both тЖТ `in_progress`)
- **Priority values**: Slate P0-P4 тЖФ Notion High/Medium/Low

### Conflict Resolution

When both sides change the same field between syncs:
1. **Detection**: Compare Slate events timestamps vs Notion `last_edited_time`
2. **Auto-resolve**: Last-write-wins (most recent timestamp wins)
3. **Manual override**: `slate notion resolve <id> --prefer local|notion`

### Sync Records

The `notion_sync` table tracks which Slate tasks are linked to which Notion pages. Cascade-deletes when the Slate task is deleted.

### Parent Relations

Parents are **bidirectional** тАФ assigned in either Slate or Notion and synced both ways. Push uses two-pass (parents first), pull resolves recursively.

See [Notion Sync Guide](notion-sync.md) for full documentation.
